# document_ops 模块修改日志

> 分支: `feat/retriever-lhh`
> 日期: 2026-04-11
> 修改文件: `operation_parser.py` / `docx_operations.py` / `common_operations.py`

---

## 修改背景

"最后一段"类表达解析失败后，执行层默认回退到"第1段"，导致误删用户真正想操作的段落。本次修改从**规则层 → 执行层**全链路修复，并补强了缓存一致性和参数归一化逻辑。

---

## 一、operation_parser.py — 规则提取增强

### 1. 位置表达规则补全

**修复前**：`_extract_parameters_by_rules` 仅能识别 `第[一二三四五六七八九十\d]+[段节条章行]` 格式。

**修复后**：按优先级分三层兜底。

| 优先级 | 匹配模式 | 提取结果 |
|--------|---------|---------|
| 最高 | `(最后|末尾|文末)(一?)段/节/条/章/行` | `position = '最后一段'` |
| | `最后那段 / 最后那一段 / 末尾那段 / 文末那段` | `position = '最后一段'` |
| 次高 | `(开头|最前面\|第一)(一?)段/节/条/章/行` | `position = '第一段'` |
| 兜底 | `第[一二三四五六七八九十百两\d]+[段节条章行]` | 原样提取 |

### 2. 中文数字正则增强

原正则缺少对 `十`、`百`、`两` 的支持，已补全。

```diff
- r'第[一二三四五六七八九十\d]+[段节条章行]'
+ r'第[一二三四五六七八九十百两\d]+[段节条章行]'
```

### 3. ADD_PARAGRAPH 专属位置提取

针对 `在第3段后添加` 类指令，新增正则捕获：

```python
after_match = re.search(r'在第[一二三四五六七八九十百两\d]+[段节条章行]后', instruction)
if after_match:
    params['position'] = after_match.group().replace('在', '')  # → '第3段后'
```

### 4. 引号内容提取重构

- 改为一次性 `re.findall(r'[""](.+?)[""]', instruction)` 提取所有引号内容
- `REPLACE_TEXT`：取第0、1个引号为 old/new
- `ADD/EDIT_PARAGRAPH`：取末个引号为 new_content
- 兼容"添加/新增/插入"后的引号内容提取为 `content`

## 二、docx_operations.py — 执行层安全加固

### 1. 新增 `_invalidate_cache()`

文档有改动后清除段落/表格缓存，避免连续操作读到旧列表：

```python
def _invalidate_cache(self):
    self._paragraphs = None
    self._tables = None
```

所有写操作（`edit_paragraph`、`format_paragraph`、`add_paragraph`、`delete_paragraph`）均在 `save` 后调用。

### 2. 新增 `_remove_paragraph_element()`

封装 XML 节点删除逻辑，`delete_paragraph` 改用此方法：

```python
def _remove_paragraph_element(self, para) -> None:
    para_element = para._element
    parent = para_element.getparent()
    if parent is not None:
        parent.remove(para_element)
```

### 3. `_find_paragraph_by_position` 移除静默 fallback

**修复前**：匹配失败时返回 `(0, paras[0])`，静默回退到第一段。

**修复后**：直接 `return -1, None`，调用方必须处理此情况。

```python
# 修复前
return 0, paras[0]

# 修复后
return -1, None  # 不再 fallback
```

### 4. `add_paragraph` 场景补全

| 场景 | 匹配词 | 插入方式 |
|------|--------|---------|
| 结尾 | `结尾`、`末尾`、`最后`、`文末` | `doc.add_paragraph(content)` |
| 开头 | `开头`、`最前`、`第一段/节/条` | `first_para.insert_paragraph_before(content)` |
| 指定段后 | `第X段后`（含"后"字） | `ref_para.insert_paragraph_before(content)` |
| 其他 | — | 默认追加到末尾 |

### 5. 写操作后统一清缓存

以下操作在 `doc.save()` 后均追加 `self._invalidate_cache()`：
- `edit_paragraph`（精确替换 / 整段替换）
- `format_paragraph`
- `add_paragraph`
- `delete_paragraph`

## 三、common_operations.py — 执行器防误删

### 1. `_normalize_docx_params` 扩展到所有操作类型

**修复前**：仅处理 `FORMAT_PARAGRAPH` 场景，其他操作类型参数归一化逻辑缺失。

**修复后**：所有 docx 操作类型均执行以下归一化。

#### 位置字段补全（多来源兼容）

```python
# 从 paragraphs[0].index 补
if not normalized.get('position') and first.get('index') is not None:
    normalized['position'] = f"第{int(first['index'])}段"

# 从顶层 index 补
if not normalized.get('position') and normalized.get('index') is not None:
    normalized['position'] = f"第{int(normalized['index'])}段"

# 从 LLM 偶发字段名补
for key in ('target', 'location', 'paragraph', 'paragraph_position'):
    val = normalized.get(key)
    if isinstance(val, str) and val.strip():
        normalized['position'] = val.strip()
        break
```

#### 位置表达归一

```python
pos = str(normalized.get('position') or '').strip()
if pos:
    if any(word in pos for word in ('最后', '末尾', '结尾', '文末')):
        normalized['position'] = '最后一段'
    elif any(word in pos for word in ('开头', '最前', '第一')):
        normalized['position'] = '第一段'
```

### 2. `_require_position()` 函数化 + 三处调用均校验

**修复前**：`edit_paragraph`、`format_paragraph`、`delete_paragraph` 三处默认 `params.get('position', '第1段')`，缺失参数时直接操作第一段。

**修复后**：统一用 `_require_position()` 校验，缺失则返回明确错误：

```python
def _require_position() -> Optional[str]:

if not position:
    return {'success': False, 'message': '缺少段落位置参数，无法确定要编辑哪一段'}

# format_paragraph
position = _require_position()
if not position:
    return {'success': False, 'message': '缺少段落位置参数，无法确定要格式化哪一段'}

# delete_paragraph
position = _require_position()
if not position:
    return {'success': False, 'message': '缺少段落位置参数，无法确定要删除哪一段'}
```

## 四、operation_parser.py — 深层 bug 修复（2026-04-11 第二轮）

### 1. `_rule_based_match` 中文 `.lower()` 导致正则失效（最严重）

**根因**：`instruction.lower()` 把所有汉字转成 Unicode 规范化形式，与正则里的大写 `"段"` 字符无法匹配——导致所有操作类型匹配全部失败。

```python
# 修复前（错误）
instruction_lower = instruction.lower()
if re.search(pattern, instruction_lower):   # 中文 .lower() 后字符已变

# 修复后
text = instruction.strip()                   # 直接用原字符串
if re.search(pattern, text):
```

### 2. 操作类型模式要求"段落"但指令只有"段"

**根因**：`OPERATION_PATTERNS` 中 `r"删除.*段落"` 要求"段**落**"两字，但用户指令"删除最后一段"只有"段"，正则永远不匹配，被迫走 LLM 分支，LLM 返回了错误的段落索引。

| 操作类型 | 修复前模式 | 修复后模式 |
|---------|---------|---------|
| DELETE_PARAGRAPH | `r"删除.*段落"` | `r"删除.*段"`, `r"删掉.*段"` |
| EDIT_PARAGRAPH | `r"编辑.*段落"` | `r"编辑.*段"`, `r"编辑.*段落"` |
| ADD_PARAGRAPH | `r"添加.*段落"` | `r"添加.*段"`, `r"添加.*段落"` |

### 3. `parse()` 置信度硬截断导致规则参数不完整

**根因**：`confidence >= 0.8` 时直接返回规则结果，但规则层不产 `bold/color`；`confidence < 0.8` 时走 LLM 但规则层的 `position` 被完全丢弃。

**修复后**：始终并行调用规则层和 LLM，合并参数——position 以规则为主（稳），bold/color 等格式参数以 LLM 为主（准）。

```python
# 修复后逻辑
rule_params  = _extract_parameters_by_rules(...)   # 始终调用
llm_result   = _llm_assisted_parse(...)           # 始终调用

# 合并：格式参数用 LLM 的，position 优先用规则的
merged = dict(llm_params)
if not merged.get('position') and rule_params.get('position'):
    merged['position'] = rule_params['position']
```

### 4. 颜色提取正则无法匹配"标为红色"等表达

**根因**：`'红色' in instruction` 要求"红色"两字紧邻，无法匹配"标为红色"、"设为红色"等中间带词的情况。

```python
# 修复后：支持"标为/设为/设置成/设置/改成/改为/为"等中间词
color_pattern = r'(?:标为|设为|设置成|设置为|改成|改为|为)?(红|蓝|绿|黑|白|黄)(色)?'
color_match = re.search(color_pattern, instruction)
if color_match:
    key = color_match.group(1) + (color_match.group(2) or '')
    params['color'] = color_map[key]
```

## 五、common_operations.py — 深层 bug 修复（2026-04-11 第二轮）

### 变量遮蔽：内层函数 `position` 与外层变量同名

**根因**：`_require_position()` 内层函数和外部 `position` 变量重名，Python 闭包捕获的是**函数对象**而非其返回值，传给 `format_paragraph(position=position)` 的实际是函数而非字符串，导致取 position 时异常，最终落到第一段。

```python
# 修复前（错误）
def _require_position() -> Optional[str]:
    position = str(params.get('position') or '').strip()   # 遮蔽外层 position
    return position or None

elif op_type == OperationType.FORMAT_PARAGRAPH:
    position = _require_position()   # position 被内层函数遮蔽
    return self.ops.format_paragraph(
        position=position,            # 传的是函数对象！
        ...
    )

# 修复后
def _require_pos() -> Optional[str]:   # 改名
    pos = str(params.get('position') or '').strip()
    return pos or None

elif op_type == OperationType.FORMAT_PARAGRAPH:
    pos = _require_pos()               # pos 是字符串
    return self.ops.format_paragraph(position=pos, ...)
```

## 六、第三轮架构重构：LLM 带文档上下文解析（2026-04-11）

### 1. 根因：LLM 在真空中解析 position

**问题**：`parse_operation(instruction, file_type)` 调用 LLM 时，LLM 完全不知道文档里有多少段落。
看到"删除第12段"就瞎猜，文档实际只有 8 段也返回 12。

```
修复前：
  API → parse_operation("删除最后一段")
    → LLM prompt: "用户指令: 删除最后一段"
    → LLM 不知道文档有多少段 → 瞎猜 position=12
    → _find_paragraph_by_position("第12段") → 越界 → 误删第一段
```

### 2. 执行层修正 position 越界

**文件**：`common_operations.py` — `OperationExecutor._resolve_position()`

执行操作前，用文档真实段落数修正 LLM 返回的 position：

```python
def _require_pos():
    raw = params.get('position', '')
    resolved = self._resolve_position(raw)   # 验证是否越界
    if resolved and resolved.isdigit():
        n = int(resolved)
        if n > para_count:
            # 越界 → 自动修正为最后一段，并打警告日志
            logger.warning(f"LLM指定第{n}段，但文档只有{para_count}段，修正为最后一段")
            return f"第{para_count}段"
    return resolved
```

### 3. LLM 解析时带上文档段落列表

**文件**：`operation_parser.py` — `OperationParser._read_docx_context()` + `_llm_assisted_parse()`

解析前先读文档结构，注入 LLM prompt：

```python
# parse() 中：
document_context = self._read_docx_context(file_path)  # 读取段落列表

# _llm_assisted_parse() prompt 中注入：
ctx_block = f"""
该文档共有 N 个非空段落，编号从1开始。
段落列表：
  第1段 [Heading 1]: 「文档标题」
  第2段 [Normal]: 「第一段内容...」
  ...

重要：你返回的 position 必须在1到N之间。
"""
```

**效果**：LLM 看到文档实际只有 5 段，就不会返回"第12段"了。

### 4. API 层传入 file_path

**文件**：`api/document_ops.py`

```python
# 修复前
operation = parse_operation(body.instruction, record.file_type)

# 修复后
operation = parse_operation(body.instruction, record.file_type, file_path=record.file_path)
```

### 5. PDF 不支持提示

**文件**：`api/document_ops.py` — `operate_document` / `batch_operations` / `preview`

对 PDF 文件类型返回明确提示：

```python
if record.file_type == "pdf":
    return {"success": False, "message": "PDF 格式暂不支持文档操作，请先将 PDF 转换为 Word 文档后再试。"}
```

## 七、第七优先级优化（2026-04-11）

### 1. parse() 改为 always-merge，不再 winner-take-all

**文件**：`operation_parser.py` — `OperationParser.parse()`

```python
# 修复前：winner-take-all，LLM 置信度高就连 position 一起用 LLM 的
if llm_confidence > rule_confidence:
    return merged_params(llm_params)

# 修复后：always-merge，position 永远以规则为准
merged_params = {}
# position：永远以规则为准
if rule_params.get('position'):
    merged_params['position'] = rule_params['position']
elif llm_params.get('position'):
    merged_params['position'] = llm_params['position']
# 格式参数：规则和 LLM 谁有谁补
for key in ('bold', 'italic', 'font_size', 'color', ...):
    if rule_params.get(key) is not None:
        merged_params[key] = rule_params[key]
    elif llm_params.get(key) is not None:
        merged_params[key] = llm_params[key]
# 操作类型：取置信度高者
```

## 八、第八轮：段落定位全面重构（2026-04-11）

### 1. 段落数据结构重构

**文件**：`docx_operations.py` — `WordDocumentOperations`

新增 `_paragraph_map` 缓存 + `_build_paragraph_map()` + `_get_paragraph_map()`：

```python
# 段落映射结构
{
    'user_index': 1,      # 用户视角编号（从1开始，连续，过滤空段落）
```

关键保证：用户视角编号和前端展示完全一致。

### 2. _find_paragraph_by_position() 三层优先级

| 优先级 | 方式 | 示例 |
|--------|------|------|
| L1 | 固定位置词 | 最后一段、开头、末尾 |
| L2 | 标准编号 | 第3段、第三段、第12段 |
| L3 | 内容兜底 | 包含"关键词"的那段 |
| L3 | 相对位置 | 第3段后面、第2段前面 |

找不到时不回退到第1段，返回 `(-1, None)`。

### 3. 规则层放宽

**文件**：`operation_parser.py` — `_extract_parameters_by_rules()`

新增位置表达支持：
- 相对位置：第3段后面 / 第2段之前
- 内容兜底：包含"X"的那段、有X的段落
- 关键词定位："X那段"（无连词）

LLM prompt 同步更新，支持关键词定位和相对位置说明。

### 4. _read_docx_context() 与段落映射对齐

编号规则与 `_get_paragraph_map()` 完全一致：
- 只统计 `text.strip()` 非空的段落
- `user_index` 从1开始、连续递增
- note 明确说明："第N段"中的N就是这个编号

### 5. 所有写操作后清缓存

`add_paragraph()`、`edit_paragraph()`、`delete_paragraph()`、`format_paragraph()` 都在保存后调用 `_invalidate_cache()`，下一次操作使用最新段落映射。

### 6. edit_paragraph/delete_paragraph 返回的段号修正

`idx` 已经是 `user_index`（1-based），不再 `+1`：
- 修复前：`已删除第{idx+1}段`
- 修复后：`已删除第{idx}段`

## 修改影响分析

| 操作 | 修复前风险 | 修复后行为 |
|------|----------|---------|
| "删除最后一段" | 规则漏匹配 → 回退第1段 → 误删 | L1固定位置识别 → 删除最后一段 |
| "删除第X段"（X超出范围） | 回退第1段 → 误删 | 执行层检测越界 → 返回失败，要求预览确认 |
| "删除包含'北京'的那段" | 规则不识别 → 无position → 回退第1段 | L3内容兜底识别 → 精确匹配关键词段落 |
| "第3段后面删除" | 规则不识别相对位置 | L3相对位置识别 → 删除第3段后那段 |
| 连续两次删除不同段落 | 缓存未失效 → 索引错位 | 每次写操作清缓存 → 段落映射始终最新 |
